# 蒙古语多模态情感与情感陪伴设计

把**蒙古语多模态情感分析**融合进个人空间智能物联助手，并以 Yanshee 机器人为载体做**情感陪伴**的设计文档。本文是动代码前的总设计，确定形态、边界和分阶段路线。

配套文档：[`docs/yanshee-integration.md`](yanshee-integration.md)（机器人接入）、[`docs/safety-policy.md`](safety-policy.md)（策略）、[`docs/device-connection-interface.md`](device-connection-interface.md)（设备事件）、[`docs/space-settings.md`](space-settings.md)（空间能力门控）。

## 1. 目标与核心原则

- **蒙古语是特色，但不是唯一。** 系统对中文、英文用户照常可用，蒙古语是被**重点建设、可对外展示**的差异化能力，而不是把其它语言换掉。
- **多模态。** 同时用**面部表情、语音韵律、文本语义**三路信号判断情绪，单一模态缺失时仍能降级工作。
- **陪伴而非操控。** 机器人作为情感陪伴宠物，目标是理解与回应情绪、给予陪伴，**绝不利用检测到的情绪做诱导或操控**。
- **长在现有平台上。** 复用平台已有的设备事件流、空间能力门控、策略引擎、审计和模型厂商配置，不另起炉灶。

### 关键设计：用"语言依赖性"自然实现"特色但不唯一"

把三个模态按是否依赖语言切开，矛盾就消解了：

| 模态 | 依赖语言？ | 作用定位 |
|---|---|---|
| **视觉**（面部表情 → 情绪） | ❌ 语言无关 | **共享底座**，服务所有用户 |
| **语音韵律**（音调/能量/语速 → 情绪） | ❌ 基本语言无关 | **共享底座**，服务所有用户 |
| **文本**（语音转写 → 文本情感） | ✅ 依赖语言 | **按语言分支**，蒙古语是被重点建设的一等分支，与中/英并列 |

→ 视觉 + 韵律构成对谁都通用的"不只蒙古语"底座；文本模态按语言分支，**蒙古语分支是特色与研究重点**。

## 2. 总体架构

```text
                 ┌──────────────── Yanshee 机器人（边缘）─────────────────┐
  用户            │  摄像头 ─→ 面部表情情绪 (FER)        ┐                │
 （表情/         │  麦克风 ─→ 语音韵律情绪 (SER)         ├─→ 边缘预处理   │
   说话） ──────→ │  麦克风 ─→ ASR 转写 ─→ 语种识别 ─→    ┘   只输出结构化  │
                 │                         文本情感(按语种分支 zh/en/mn) │
                 └──────────────┬─────────────────────────────────────┘
                                │  仅结构化情绪结果（不传原始音视频）
                                ▼   POST /api/device-connections/{id}/events
                 ┌──────────────────────── 平台后端 ──────────────────────┐
                 │  情感融合引擎 (late-fusion + 时序平滑)                  │
                 │     → 情绪状态（primary / valence / arousal / 语种）    │
                 │  陪伴行为策略：情绪 → 回应策略                          │
                 │     → 豆包 LLM 生成共情话语（按用户语言）               │
                 │     → policy.py 门控动作（安全手势集，确认/审计/限速）  │
                 │  情绪事件流 / 宠物人格&记忆 / 审计                       │
                 └──────────────┬─────────────────────────────────────────┘
                                ▼  受控回应
                 Yanshee：TTS（用户母语，含蒙古语）+ 安全手势（点头/歪头/伸手）
```

边缘做感知、只上传结构化结果；服务器做融合、决策、生成和门控；机器人做受控回应。

## 3. 三模态设计

| 模态 | 输入 | 方法（v0 → 进阶） | 运行位置 | 语言 |
|---|---|---|---|---|
| 视觉 FER | 摄像头帧（1–2s 采样） | 轻量人脸表情分类（7 类 + valence/arousal）→ 研究级模型 | 边缘优先；Pi 3B 算力不足时可降级到服务器 | 无关 |
| 语音 SER | 麦克风音频窗 | 韵律特征(openSMILE)+分类器 / wav2vec2-SER | 边缘或服务器 | 基本无关 |
| 文本情感 | ASR 转写文本 | 语种识别 → 分语种文本情感分类 | 服务器 | **依赖** |

文本分支：
- **中文 / 英文**：用成熟多语种情感模型。
- **蒙古语（特色）**：用蒙古语文本情感模型（参考 2025 年 GTAH 等研究），必要时在公开资源上微调。**这是研究与展示重点。**
- ASR 同理分语种：中/英成熟，**蒙古语 ASR 是难点**（见第 9 节资源现状）。

## 4. 融合引擎

- **v0：晚融合（late fusion）。** 每个模态输出"情绪分布 + 置信度"，按置信度加权合并成统一情绪状态。优点：实现简单、**某一模态缺失（没说话、背光看不清脸）时自动降级**，鲁棒。
- **时序平滑。** 情绪不应每帧跳变：用短滚动窗口 + 滞回（hysteresis），输出稳定的情绪状态而非瞬时抖动。
- **进阶：研究级融合。** 升级到门控 Transformer + 自适应超模态（GTAH）、多级注意力等需要对齐的多模态蒙古语训练数据——作为研究贡献与后续迭代，不在 v0。
- **统一输出**：`primary_emotion`（happy/sad/angry/surprise/fear/disgust/neutral）+ 连续 `valence`(-1..1) / `arousal`(0..1) + `language` + 各模态分解 + 总置信度。

## 5. 情感陪伴行为回路

```text
感知（三模态）→ 融合（情绪状态）→ 理解（豆包按语言生成共情）→ 回应（TTS + 安全手势）→ 审计
```

情绪 → 回应策略（示意，可配置）：

| 检测情绪 | 回应基调 | 手势（安全集） | 语音 |
|---|---|---|---|
| 悲伤 | 安抚、陪伴 | 歪头、缓慢靠近、伸手 | 母语轻声安慰 |
| 高兴 | 共情、活泼 | 点头、招手、小舞蹈 | 母语回应 |
| 生气 | 降温、倾听 | 后撤一点、低头 | 平静共情语 |
| 平静/中性 | 轻陪伴 | 偶尔点头 | 简短回应或不打扰 |

- **豆包 LLM 只负责"把话说自然"**：情绪判定、安全检查、手势选择都由确定性逻辑完成；LLM 在结构化结果之后生成共情话语（沿用平台"工具优先、策略优先"原则）。
- **手势全程经 `policy.py` 门控**：只允许安全手势集；非平凡动作需确认；连续动作限速；全部写审计。机器人是物理执行器，按平台风险模型对待。

## 6. 大模型：豆包 Doubao（已选）

选豆包的理由：国内**响应速度最强**（约 0.3s 首字、高 token/s），陪伴对话对延迟敏感，快=体感好。

接入方式（**不改架构，加一条厂商**）：平台 `services/api/app/model_providers.py` 是声明式 `PROVIDERS` 目录，豆包走火山引擎 Ark 的 **OpenAI 兼容**接口，加一个 `ModelProvider` 条目即可：

```python
# 形态示意；接入前需核对当前准确的 base_url 与模型名（火山引擎 Ark 控制台）
ModelProvider(
    id="doubao",
    display_name="字节豆包",
    docs_url="https://www.volcengine.com/docs/82379",
    endpoints=[ProviderEndpoint(
        id="doubao_ark_openai",
        protocol=ProviderProtocol.openai,
        base_url="https://ark.cn-beijing.volces.com/api/v3",
    )],
    models=["<接入前核对>"],
    default_model="<接入前核对>",
)
```

配套：
- **流式输出（SSE）**：陪伴对话务必流式，先出字、显著改善体感；接入时一并确认 `/agent` 链路是否已流式。
- **降级与兜底**：豆包为主，保留 Kimi / 小米 MiMo 作为兜底；可加永久免费的 ERNIE-Speed 做最低降级档。
- **延迟优势叠加**：服务器在腾讯云，若后续也接腾讯混元可作同云低网络延迟的备选。

## 7. 接入平台（复用现有能力）

- **情绪即事件流。** 复用 `POST /api/device-connections/{device_id}/events`，与 `presence_detected` 同构：

```json
{
  "event_type": "emotion_detected",
  "severity": "info",
  "confidence": 0.82,
  "space_id": "space_living_001",
  "zone": "沙发",
  "attributes": {
    "primary_emotion": "sad",
    "valence": -0.4,
    "arousal": 0.2,
    "language": "mn",
    "modalities": {
      "face":  {"emotion": "sad",     "confidence": 0.78},
      "voice": {"emotion": "sad",     "confidence": 0.71},
      "text":  {"emotion": "neutral", "confidence": 0.55, "transcript_lang": "mn"}
    },
    "fusion": "late_weighted",
    "edge_model": "fer-lite + ser + mn-text-emo"
  }
}
```

只传**结构化情绪结果**，不传原始音视频。

- **空间能力门控。** `/spaces` 里"摄像头/人脸/情绪"本就是**已规划、默认关闭**的能力轴——情感分析只在 `local_only` 且显式开启情绪能力的空间运行。直接复用，不新增门控机制。
- **宠物人格与记忆。** 新增一个小存储（仿 `services/api/.local/` 现有 JSON 存储），保存宠物人格设定、最近情绪轨迹与互动记忆；保留期与 `agent_conversations` 一致（30 天），用户可手动删除。
- **智能体可读情绪。** 把"当前情绪状态"做成智能体的一个只读工具（像传感器读数），回应动作仍走策略门控。

## 8. 安全、隐私与伦理（与平台既有边界一致）

- **动作即高风险。** 情绪驱动的机器人动作 = 物理执行器输出 → `policy.py` 门控、确认、限速、审计；只允许预定义安全手势集。
- **音视频高敏感。** 边缘优先：原始音视频不出机器人、不入库、不建人脸身份库、不存原始音频；只上传结构化情绪结果。沿用平台现有隐私边界。
- **情绪是敏感个人数据。** 本地保存、限定保留期、用户可删除；不对外发送，不做画像贩卖。
- **不操控。** 系统不得利用检测到的负面情绪进行诱导、上瘾设计或商业操控；这是产品红线，应在策略与提示层显式约束。
- **提示注入防护。** 沿用 `policy.py` 既有注入检测：外部文本/用户文本不能提升动作权限。

## 9. 数据与模型资源现状（务实预期）

- **蒙古语多模态情感属研究前沿、数据稀缺**，落地需要借力公开资源 + 自建数据：
  - 研究：2025 年蒙古语多模态情感工作（GTAH 门控 Transformer + 自适应超模态、多级注意力 + 卷积增强融合）。
  - 数据：**MnTTS2**（多说话人蒙古语 TTS，约 30 小时）——可支撑蒙古语 TTS 与语音侧。
- **蒙古语 ASR 是主要瓶颈**：成熟度远低于中/英，可能需要选型 + 微调或数据采集。
- **边缘算力限制**：Yanshee 内置 Pi 3B 跑深度多模态模型吃力，视觉/语音模型可能需**降级到服务器推理**（隐私 vs 算力的权衡，见待定决策）。
- **实时性**：先做近实时（1–2s 窗口），不追求逐帧实时。

## 10. 分阶段路线

1. **本设计文档**（当前）。
2. **豆包上线**（并行快赢）：加厂商 + 确认流式，先把响应速度问题解决。
3. **语言无关底座**：面部表情 + 语音韵律 → 情绪事件，进平台、对所有用户即时可用。
4. **文本分支**：分语种 ASR + 文本情感，先中/英跑通，**接入蒙古语分支**（特色）。
5. **融合引擎 v0**：晚融合 + 时序平滑，输出统一情绪状态。
6. **陪伴回路**：情绪 → 豆包共情 → TTS（含蒙古语）+ 安全手势（策略门控）。
7. **进阶**：研究级融合模型、蒙古语 TTS 人格、长期记忆与个性化。

## 11. 待定决策（需你后续拍板）

- **视觉/语音推理放边缘还是服务器？** 边缘更护隐私、但 Pi 3B 算力弱；服务器更强、但要传音视频（可加密+即用即弃缓解）。
- **宠物人格设定**：名字、语气、主要陪伴对象与场景。
- **蒙古语深度**：仅情感识别支持蒙语，还是连 TTS 回应/界面也蒙语？
- **蒙古语数据来源**：用公开资源微调，还是需要采集/标注自建数据集？
